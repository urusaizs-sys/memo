Part 1 — 理論

賢いモデル ≠ 使えるシステム

LLMは本質的に確率的なテキスト補完エンジン。賢くなっても出力は毎回ゆらぎ、そのままではシステムに組み込めません。モデルを差し替えるのではなく、「どう使うか」を設計する3層が要ります。

  • Prompt ― どう頼むか（発話の設計）
  • Context ― 何を見せるか（情報の取捨選択）
  • Harness ― 出力をどう使うか（検証・制御の骨格）

Layer 1 · Prompt — 例

回路図ネットリスト解析

prompt.txt
// 悪い ― 出力が毎回バラバラ "このネットリストを解析してください" // 良い ― 構造化された出力が安定 あなたはEDAツールの専門家です。次のKiCad netlistから: 1. 未接続ピンを列挙 2. VCC/GNDに未接続のコンポーネントを警告 3. 出力は必ずJSON形式で返すこと 制約: 推測で補完しない / 不明は "UNKNOWN" / 説明文は不要
悪い例の出力

「解析しました。いくつか問題が…」
→ パースできない

良い例の出力

{ "unconnected": ["U1.PIN3"], "warnings": ["C2: no VCC"] }
→ 即システムに組み込める

Layer 2 · Context — 例

バイリンガル会議録の要約パイプライン

1時間の会議録 ≒ 1万トークン。全部渡すとコスト・精度が悪化する。4ステップで圧縮する。

STEP 1発話者セグメント分割 ― speaker_A / B / C に分類
STEP 2スパース化 ― 「決定 / TODO / 懸念」を含む発話のみ保持 → 約80%削減
STEP 3ローリングウィンドウ ― 直前5分のみ短期メモリに保持
STEP 4ドメイン辞書を先頭に注入 ― 略語辞書でLLMの誤解釈を防ぐ

RAG（検索拡張生成）もContext Engineeringの一形態。関連する断片だけを動的に注入します。

Superpowers — 正確性の注記

事実として押さえる点

配布: npm ではなく Claude Code のプラグインマーケットプレイス経由（/plugin marketplace add obra/superpowers-marketplace）。

スキル数: 公開情報で「12スキル」「20以上」などバージョンにより変動。具体数は断定しないのが安全です。

主なスキル名（実在）: brainstorming / writing-plans / executing-plans / test-driven-development / systematic-debugging / subagent-driven-development / requesting-code-review など。

コマンド例: /brainstorm · /write-plan · /execute-plan。

Phase 01 · Brainstorm

まず発散させる

ソクラテス的な対話で、ユーザーが「言ったこと」ではなく 「本当に欲しいもの」 を引き出し、検証済みの仕様へ昇華させるフェーズ。

ゲート: 設計に合意するまで実装には入りません（どんなに簡単に見えるタスクでも）。これが方向誤りによる手戻りを防ぎます。

コマンド: /brainstorm

Phase 02 · Branch

作業を分離する

git worktree で、現在の作業空間から隔離した環境を用意します。フィーチャー作業を本流から守り、実験や失敗を安全に切り離せます。

効果: 失敗時に安全にロールバックでき、Harness 層の「リトライ制御 / 分離」に対応します。

Phase 03 · Plan

計画を書いてから動く

タスクを小さな単位に分解し、ファイルパス・コマンド・依存関係まで明記した plan.md を作成します。チェックボックスがセッション跨ぎの状態管理になります。

コマンド: /write-plan

plan.md（状態ログ）
## Task Breakdown - [x] Task 1: JWT検証関数を追加 - [x] Task 2: ユニットテスト追加 - [ ] Task 3: ミドルウェア統合 - [ ] Task 4: E2Eテスト実行 # チェックボックスがセッション復元の手がかり
Phase 04 · Execute

規律ある実装

確定した計画をサブエージェントが実装します。実行段階は、複数のスキルが規律づけます。

TDDtest-driven-development ― RED → GREEN → REFACTOR
Debugsystematic-debugging ― 4段階・根本原因から
並列subagent-driven-development ― タスクを並列実行
Reviewrequesting-code-review ― マージ前の検証

コマンド: /execute-plan

Lens · Prompt

Prompt Engineering として

各スキル（markdown）には、最適化済みの指示・チェックリスト・問いかけの構造が書き込まれています。とくに brainstorming スキルは、ソクラテス的な対話で「本当に欲しいもの」を引き出す問いの設計そのものです。

＝「どう頼むか」を、ツール側があらかじめ用意している状態。

Lens · Context

Context Engineering として

スキルファイルはセッション開始時にコンテキストへ注入されます。さらに CLAUDE.md でプロジェクト固有の知識（スタック・規約・命名）を与えます。

＝「何をモデルに見せるか」を、ファイルという形で設計している。

Harness Engineering として

Superpowers が最も強いのがこの層です。
